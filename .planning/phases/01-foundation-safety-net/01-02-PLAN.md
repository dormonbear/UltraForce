**Mocking strategy:**
The WindowManager is a singleton class with heavy dependencies on DOM, Chrome APIs, React, and Salesforce API. Mock all external dependencies:

```typescript
vi.mock('./salesforce-api')
vi.mock('./auth', () => ({
  getSession: vi.fn(),

}))
vi.mock('./keyboard-interceptor')
vi.mock('./fuzzy-search')
vi.mock('./metadata-cache')
vi.mock('./domain-utils')
```

**Singleton cleanup:** In `afterEach`, call `destroy()` on the instance and reset static properties to prevent test contamination. The singleton pattern means tests MUST clean up or later tests see stale state.

**Test organization by domain (nested describe blocks):**

**Public API -- getInstance / lifecycle:**
- should create singleton instance
- should return same instance on subsequent calls
- should reset after destroy()
- should handle rapid getInstance calls (race condition)

**Public API -- toggle / show / hide:**
- should toggle visibility state
- should show when hidden
- should hide when visible
- should not throw when show() called while already visible
- should not throw when hide() called while already hidden

**Public API -- state management:**
- should return current state via getState()
- should update state via updateState()
- should merge partial state updates (not replace)
- should report visibility via isVisible()
- should report initialization via isInitialized()

**Public API -- event emitter:**
- should register event handler with on()
- should unregister event handler with off()
- should emit events to registered handlers
- should not emit to unregistered handlers
- should handle multiple handlers for same event

**Public API -- getDebugInfo:**
- should return debug information object
- should include host and visibility state

**URL Building (test through handleResultClick or extract):**
Since URL building methods are private, test them indirectly by:
1. Setting up the window.location mock to simulate different Salesforce domains
2. Calling handleResultClick (via the component callback mechanism) with different result types
3. Asserting chrome.tabs.create was called with the correct URL

Test cases:
- should build Lightning setup URL for standard domain
- should build Classic setup URL when preference is classic
- should handle Enhanced Domain format (*.my.salesforce.com)
- should handle sandbox domains (*.sandbox.my.salesforce.com)
- should handle Visualforce domain URLs

**Setup Shortcuts:**
- should match shortcut by partial name
- should return all shortcuts when query is empty
- should be case-insensitive

**Navigation routing:**
- should navigate to Apex class detail page
- should navigate to custom object setup page
- should navigate to Flow builder
- should navigate to user profile page
- should handle unknown result type gracefully

**Record context detection:**
- should detect record context from standard Lightning URL
- should return null for non-record pages

**User/Profile resolution (mocked sfRest):**
- should fetch current user ID via API
- should fetch current user profile ID
- should handle API errors gracefully (return null)

**Page layout / record type navigation:**
- should resolve page layout for given object and record type
- should handle missing layout assignment

If any private methods prove impossible to test through public API, use bracket notation sparingly: `(instance as any)['privateMethodName']()` -- but prefer indirect testing through public methods.

Follow project conventions: `describe/it` with "should" pattern, `beforeEach` for isolation, `it.each` for parameterized domain tests.
