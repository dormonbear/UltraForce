# Phase 03 Plan 02: Zustand Stores

**Goal:** Replace WindowManager singleton state with Zustand stores for settings, session, and search -- making state testable, predictable, and React-friendly

## Task 1: Install zustand + create stores

Install `zustand` dependency.

Create three stores in `src/stores/`:

**`settings-store.ts`** (~80 lines):
- All fields from `SearchSettings` interface (selectedTypes, shortcutKey, closeOnNavigate, etc.)
- Zustand `persist` middleware with chrome.storage adapter via storage-service
- Actions: `updateSettings(partial)`, `setNavigationMode`, `setFuzzySearch`
- `useSettingsStore` hook with selectors

**`session-store.ts`** (~40 lines):
- `sfHost`, `hasSession`, `userLightningPreference`
- Actions: `setSession(sfHost, hasSession)`, `setLightningPreference(pref)`
- No persistence (derived from auth/cookies each page load)

**`search-store.ts`** (~60 lines):
- `isVisible`, `searchResults`, `isLoading`, `searchError`, `recordContext`
- Actions: `show()`, `hide()`, `setLoading()`, `setResults()`, `setError()`, `setRecordContext()`
- No persistence (transient UI state)

Create tests for each store (~150 lines total).

## Task 2: Migrate consumers to stores

**SearchModal.tsx:**
- Replace local settings state (14 `useState` calls for settings) with `useSettingsStore`
- Replace settings loading/saving `useEffect` with store subscription
- Replace search state props (isLoading, searchResults, searchError) with `useSearchStore`
- Replace session props (sfHost, hasSession) with `useSessionStore`
- Keep callbacks as props from WM (onSearch, onClose, etc.)

**WindowManager:**
- Remove settings fields from `WindowManagerState` (closeOnNavigate, navigationMode, fuzzySearch)
- Remove settings storage listener (store handles sync)
- Update stores instead of `this.state` for search state (isLoading, searchResults, searchError)
- Update session store during init/show
- Simplify `renderComponent()` - fewer props to pass
- Read from stores (e.g. `useSettingsStore.getState()`) for navigation decisions

**salesforce-search.tsx:**
- Subscribe to settings store for shortcutKey instead of direct storage reads
- Remove storage change listener for settings

**setup-enhancer.ts:**
- Read autoLoadFields from settings store instead of direct storage read

## Success Criteria
1. Settings, session, and search state managed by Zustand stores (STAT-01)
2. No more 3-way settings duplication (WM state + SearchModal state + chrome.storage)
3. WM uses stores instead of `this.state` for settings/session/search
4. All tests pass, zero TS errors, zero lint issues
